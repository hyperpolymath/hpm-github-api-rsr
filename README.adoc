*Composite GitHub App API client for the Hyperpolymath RSR Standard.
Links and composes `hpm-crypto-rsr` (RS256 signing), `hpm-http-client-rsr`
(HTTPS), and `hpm-json-rsr` (request/response bodies) into one library
that exposes a JWT + REST surface tailored to OikosBot.*

Final integration sibling. Top of the RSR stack used by the OikosBot
AffineScript port.

== Motivation

The OikosBot bot-integration layer (currently ReScript) makes seven
distinct GitHub API calls — all sharing the same auth flow (JWT → installation
token → REST). Each call requires composing crypto + http + json + base64url.

Centralising that composition in one library means:
- AffineScript callers get a single typed surface (no inline base64url, no
  inline JSON construction)
- The auth-flow logic stays in one place (testable, reviewable)
- The four primitives stay focused and reusable

See link:ABI-FFI-README.md[ABI-FFI-README.md] for the full C ABI surface.

== Dependencies

Build-time links against three sibling RSR libraries:

* https://github.com/hyperpolymath/hpm-crypto-rsr[hpm-crypto-rsr] — RS256 signing
* https://github.com/hyperpolymath/hpm-http-client-rsr[hpm-http-client-rsr] — HTTPS
* https://github.com/hyperpolymath/hpm-json-rsr[hpm-json-rsr] — JSON parse + escape

The sibling `libhpm_*.so` files must be on the linker path at build time
and on `LD_LIBRARY_PATH` at runtime (or installed system-wide).

== Build

[source,shell]
----
# Build siblings first (one-off):
( cd ../hpm-crypto-rsr/ffi/zig       && zig build )
( cd ../hpm-http-client-rsr/ffi/zig  && zig build )
( cd ../hpm-json-rsr/ffi/zig         && zig build )

cd ffi/zig
zig build           # builds libhpm_github_api.{so,a}
zig build test      # runs Zig test suite (offline JWT + parsing)
----

== Lifetime model

`hpm_github_token_t*` — heap-allocated wrapper holding a NUL-terminated
token string. Free with `hpm_github_token_free`.

`hpm_github_response_t*` — heap-allocated wrapper holding status code +
response body bytes. Free with `hpm_github_response_free`.

Both are independent allocations; child handles are not a thing here.
